Part 2: Interesting SVG examples

Find three interesting examples of the usage of SVG on the Web. Offer a brief description of each and their URL. Let me know if I can share your recommendations with others.

Feel free to share these examples. They are all examples I discovered before commencing the course:

  1. OpenStreetMap's Osmarender

    This is not rendered as SVG, but rather uses SVG to create maps from OpenStreetMap (OSM) data, and creates rasterised tiles for serving to the browser (using an interesting distributed processing system like SETI@home). I have actually hacked on Osmarender a little myself. I don't particularly like the way the XSLT takes the rule file as input rather than the data, and would like to refactor it to use a more natural input (like the, er, map data). The project inspired me to look into what else might be done with SVG and maps, particularly interactivity, and lead me to the next examples. I find it interesting that there is a debate about when to render curves as beziers, and how to declare preferences.

  2. Yosemite National Park - Interactive Hiking Map

    This the sort of place I hope Osmarender might get to in the browser without post-rasterising. It's an amazingly inspiring interactive map of Yosemite National Park. I'm not quite sure from where it's sourcing its geocoordinates (there's a paper written about it, so with more time I can find out), but I can see the park boundary hardcoded as an svg path.

    I particularly like the "Animate Trail!" link in the trails tab, which cleverly degrades to "Show Trail!" on Firfox, which doesn't support animation. Calculating the profile adjacent to this link is just brilliant and I can imagine vetry important. Wow!

  3. Mozilla SVG Tetris

    This is the old Tetris game implemented in SVG. It gets by without declarative animation (source), so works in just about any static SVG-supporting browser platform. Using Javascript timers and event listers to reposition the tiles also means we get the retro jerkiness of movement that the classic version had. I daresay a declaratively animated version would have produced quite smooth movements. In fact it's almost completely implemented with Javascript, with only three SVG elements after <script/>, two of which are near-empty group elements (<g/>) acting as scaffold.

    It's curious that something implemented in Javascript, undoubtedly less accessible in theory, is in practice able to be appreciated in full fuctionality by a greater audience because of incomplete SVG standards support.
